# Barí — Project Architecture

## Overview

Barí is a self-contained WordPress development framework built around Docker. Its goal is *digital sovereignty*: providing teams with a fully controlled, reproducible stack that can be deployed on local hardware or private cloud environments, free from dependency on centralised SaaS services.

The project is structured as a **monorepo** with clean separation between infrastructure configuration, automation tooling, frontend source code, and the WordPress application.

## Key Architectural Decisions

### 1. WordPress Core Isolation (Bedrock-like)
WordPress core files live in `app/wp/` rather than at the web root. Nginx rewrites `wp-admin`, `wp-includes`, and `wp-*.php` paths to `/wp/`, keeping core files separate from application code in `app/web/`. This makes upgrades safer and version control cleaner.

### 2. Composer-Managed Dependencies
All third-party plugins and themes are installed and updated via Composer using [WPackagist](https://wpackagist.org) as the package repository. No manual plugin downloads. Premium plugins (e.g. ACF Pro) can be served via a private [Satispress](https://www.satispress.com) instance.

### 3. Environment-Driven Configuration
Every runtime constant — database credentials, WordPress settings, SMTP, memory limits, debug flags — is injected via environment variables from `.env`. The Docker image reads these and writes `wp-config.php` automatically. There is no hardcoded configuration in the codebase.

### 4. Timber/Twig Templating
The custom theme uses [Timber](https://timber.github.io/docs/) to bridge WordPress's PHP template system with [Twig](https://twig.symfony.com/) templates. PHP Hook classes handle WordPress integration; Twig handles all HTML output.

### 5. Container-First CLI
All day-to-day commands (Composer, WP-CLI, pnpm) are proxied through shell scripts in `bin/` that transparently route to the correct Docker container. Developers never need to type raw `docker exec` commands.
