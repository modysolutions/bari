# Frontend Build

Frontend assets are compiled using **`@wordpress/scripts`** (wp-scripts), which wraps Webpack 5 with a WordPress-optimised default configuration. The setup is extended in `config/webpack/webpack.config.js`.

The build pipeline is **centralised at the project root** — all custom styles and scripts for the theme (and any plugin that uses them) are authored in `src/` and compiled to a single output directory. No individual theme or plugin maintains its own build pipeline.

---

## Entry Point

| Entry | Source | Output | Purpose |
|---|---|---|---|
| `theme` | `src/app.js` | `app/dist/theme.js` + `theme.css` | All theme frontend assets |

`src/app.js` imports `src/app.scss` (and any other JS modules). The SCSS is extracted into a separate CSS file by the MiniCSSExtractPlugin included in wp-scripts defaults.

---

## Output Location

All compiled assets are written to **`app/dist/`**, which is web-accessible at `https://{host}/dist/`.

| File | Purpose |
|---|---|
| `dist/theme.js` | Compiled theme JavaScript |
| `dist/theme.css` | Compiled theme CSS |
| `dist/theme.asset.php` | Version hash + dependency list (used by PHP enqueue) |

The `.asset.php` file is generated by `@wordpress/dependency-extraction-webpack-plugin` and is used in `App.php` and `Gutenberg.php` to enqueue assets with automatic cache-busting.

---

## npm Scripts

Defined in `package.json` at the project root:

```bash
# Development (watch mode, source maps, fast rebuilds)
pnpm start

# Production build (minified, optimised, no source maps)
pnpm build
```

Both commands use the custom webpack config via `--config-file=config/webpack/webpack.config.js` and write output to `app/dist/` via `--output-path=app/dist`.

**Package manager:** `pnpm@10.14.0`

## Webpack Configuration

`config/webpack/webpack.config.js` extends the wp-scripts defaults:

```js
const path    = require('path');
const defaults = require('@wordpress/scripts/config/webpack.config.js');

module.exports = {
    ...defaults,
    entry: {
        theme: path.resolve(process.cwd(), 'src', 'app.js'),
    },
    module: {
        ...defaults.module,
        rules: [
            ...defaults.module.rules,
            // Images → copied to dist/ as static assets
            {
                test: /\.(png|svg|jpg|jpeg|gif)$/i,
                type: 'asset/resource',
            },
            // JS/JSX → Babel with React preset
            {
                test: /\.(js|jsx)$/,
                exclude: /node_modules/,
                use: {
                    loader: 'babel-loader',
                    options: { presets: ['@babel/preset-react'] },
                },
            },
        ],
    },
};
```

Additional module rules:
- **Images** (`png|svg|jpg|jpeg|gif`) — Handled as `asset/resource` (copied to `dist/`)
- **JS/JSX** — Processed via `babel-loader` with `@babel/preset-react`

## SCSS Architecture

Source styles live in `src/scss/` and are imported by `src/app.scss`.

```
src/scss/
├── helpers/
│   ├── _variables.scss       # Design tokens (colours, spacing, breakpoints)
│   ├── _functions.scss       # SCSS functions
│   ├── _mixins.scss          # Reusable mixins
│   ├── _normalize.scss       # CSS reset/normalise
│   ├── _media-queries.scss   # Breakpoint mixins
│   ├── _container.scss       # Max-width container utility
│   └── _fonts.scss           # @font-face declarations
├── layout/
│   ├── _base.scss            # Global element styles (body, links, headings)
│   ├── _typography.scss      # Typography scale
│   └── _editor-base.scss     # Editor-specific layout overrides
└── components/
    └── _button.scss          # Button component
```

### `src/app.scss`

Imports helpers, layout, and component partials for the frontend. All custom theme styles belong here — do not add styles directly to the theme's `assets/` directory or to individual plugin stylesheets.

---

## WordPress Block Styles

`Gutenberg.php` declares `add_theme_support('wp-block-styles')`, which loads WordPress's default block CSS. This ensures core blocks (buttons, images, separators, etc.) look correct without requiring custom styles for each.

---

## Adding New JS Modules

Add imports to `src/app.js`:

```js
import './modules/slider';
import './modules/accordion';
```

Or create separate entry points in `webpack.config.js` for code-split bundles (e.g. for a page-specific heavy library).

## Design Tokens → CSS Custom Properties

`theme.json` generates CSS custom properties that are available both in SCSS (via `var()`) and in the Gutenberg editor:

```css
/* Generated by theme.json */
--wp--preset--color--primary:     #005f6b;
--wp--preset--font-size--large:   clamp(1.25rem, 2vw, 1.75rem);
--wp--preset--spacing--md:        1.5rem;
--wp--preset--font-family--sans:  'Sansation', sans-serif;
```

Reference these in SCSS to keep frontend and editor styles in sync:

```scss
.my-element {
    color: var(--wp--preset--color--primary);
    padding: var(--wp--preset--spacing--lg);
}
```
